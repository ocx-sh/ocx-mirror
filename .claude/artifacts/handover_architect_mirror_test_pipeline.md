## Problem

OCX mirrors upstream binaries into OCI registries. Currently there is no pre-publish validation that a mirrored package works on its declared platforms. We want: per-repo CI in `ocx-sh/mirror-<tool>` repos that tests each new (version, platform) tile before push to the canonical registry.

## Scope & Phasing

### Phase 1 (this design)

- Linux platforms (amd64, arm64) tested in multiple docker base images (e.g. ubuntu, alpine, fedora) on GHA ubuntu host runners
- macOS (amd64 via Rosetta, arm64 native) tested natively on macos-latest
- Windows (amd64) tested natively on windows-latest with pwsh
- Reporting: Discord webhook per mirror repo, silent default
- Distribution: ocx-mirror via `cargo install --git ... --rev <SHA>` pinned in mirror.yml

### Phase 2 (deferred, not in scope)

- Foreign archs (mips, sparc, ppc64le, s390x, riscv64) via qemu-in-docker with ocx-in-container
- ocx-mirror as OCX package (dogfood install)
- Aggregator workflow in central repo for org-wide dashboard
- Wine for windows-on-linux (researcher flagged unreliable for MSVC binaries)

## User-Confirmed Decisions

These have been settled through the sparring rounds. Architect should reaffirm or challenge each — do not silently accept.

### D1: Repo-per-mirror under `ocx-sh/mirror-<tool>`

One GitHub repo per mirrored tool, same `ocx-sh` org. No separate `ocx-mirror` org.

- Pro: per-tool CODEOWNERS, isolated breakage, single billing
- Con: 100s of repos at maturity; needs fan-out tooling for ocx-mirror version bumps

### D2: Renderer subcommand `ocx-mirror generate ci`

cargo-dist analog: reads `mirror.yml`, emits `.github/workflows/*` (+ scripts only if needed). Generated files carry header `# DO NOT EDIT — generated by ocx-mirror generate ci`. `--check` mode flags drift.

### D3: Reuse existing `ocx package test` for smoke execution

No new `ocx package` subcommand. Executor logic (qemu / rosetta / docker) lives in the **test command itself** (the part after `--` in `ocx package test`), wired into the rendered workflow step. Not an ocx concept.

### D4: Static matrix from mirror.yml + dynamic version list inside each leg

Matrix shape (platforms × containers) is known at render time and static. Versions list is runtime-discovered via `discover` job's `versions` JSON output. All matrix legs gated on `discover.has_new`. Each leg loops over `versions` internally.

Rationale: PR check names stay stable, `required_status_checks` branch protection works, zero-cost no-op runs when nothing new.

### D5: Pipeline shape

```
discover (ubuntu)
  └─> ocx-mirror plan
      outputs.has_new: bool
      outputs.versions: []          # JSON list of new versions

prepare (matrix per version, if has_new)
  └─> ocx-mirror prepare --version <V>
      uploads bundle-{V}-{P}.tar.xz artifacts (all platforms)

smoke-and-push (matrix per platform×container STATIC from mirror.yml, runs-on per matrix entry, if has_new)
  └─> for V in versions:
        download bundle-{V}-<this-platform>
        ocx package test --platform <P> bundle.tar.xz -- <docker-or-native test cmd>
        if green: ocx package push --platform <P>
        record result-{V}-{P}-{runner-id}.json

cascade (ubuntu, if has_new)
  └─> aggregate results across all tiles
      per-version status: published | partial | failed
      cascade tags per pushed-platforms-set
      promote latest only on fully-green newest
      emit run-summary.json

notify (if has_new AND (any-new-green OR any-red))
  └─> read run-summary
      post Discord webhook
```

### D6: Test config shape in mirror.yml

```yaml
test:
  test: ./tests/smoke.sh             # auto-detect: filepath -> script, else command
  # OR explicit:
  # script: ./tests/smoke.sh
  # command: "cmake --version | grep ${OCX_VERSION}"
  working_dir: "${install_dir}"      # optional, default = install dir

platforms:
  linux/amd64:
    runner: ubuntu-latest
    containers:
      - { image: "ubuntu:24.04", shell: bash }
      - { image: "alpine:3.20",  shell: sh }
      - { image: "fedora:40",    shell: bash }
  linux/arm64:
    runner: ubuntu-24.04-arm
    containers:
      - { image: "ubuntu:24.04", shell: bash }
      - { image: "alpine:3.20",  shell: sh }
  darwin/arm64:
    runner: macos-latest
  darwin/amd64:
    runner: macos-latest
    prefix: ["arch", "-x86_64"]      # optional rosetta wrapper
  windows/amd64:
    runner: windows-latest
    shell: pwsh
    # per-platform test override allowed:
    test:
      script: ./tests/smoke.ps1
```

Auto-detect rule for the single-field `test:` form:

| Input pattern | Detected as |
|---|---|
| Starts with `./`, `/`, `scripts/`, `tests/` | script |
| Matches `\.(sh\|ps1\|py\|bat)$` | script |
| Path exists in mirror repo | script |
| Otherwise | command |

Env exposed at test time: `OCX_INSTALL_DIR`, `OCX_VERSION`, `OCX_PLATFORM`, `OCX_IMAGE`.

### D7: Multi-runner per platform via `containers: []`

Linux platforms specify `containers:` array — each entry generates a matrix row. Mac/Windows omit `containers:`, single native entry. (V, P) tile green only if **all** runners/containers green (AND across the runner axis).

### D8: ocx-in-container execution model for linux

Docker image IS the execution host on linux. ocx runs **inside** the container. GHA runner is just a launcher.

- Native arch alignment for phase 1: linux/amd64 → amd64 image, linux/arm64 → arm64 image. No qemu in phase 1.
- Phase 2 adds qemu layer below docker for foreign archs. Same ocx-in-container shape — qemu inserted underneath. This keeps phase 1 → phase 2 transition minimal.
- Mac/Windows: native runner, ocx native. No container layer.

Rendered linux smoke step (sketch):

```bash
docker run --rm \
  -v "$RUNNER_TEMP/bundle.tar.xz:/bundle.tar.xz:ro" \
  -v "$GITHUB_WORKSPACE:/workspace:ro" \
  -w /workspace \
  -e OCX_VERSION="$VERSION" \
  -e OCX_PLATFORM=<P> \
  -e OCX_IMAGE=<image> \
  <image> <shell> -c '
    <install-ocx-in-container>     # see A11
    ocx package test --platform <P> /bundle.tar.xz -- <user-test-cmd>
  '
```

### D9: Distribution via pinned-SHA cargo-install

```yaml
# mirror.yml
ocx_mirror:
  rev: abc123def...        # pinned SHA in github.com/ocx-sh/ocx
```

CI fragment: `cargo install --git https://github.com/ocx-sh/ocx --rev $REV --bin ocx_mirror`. Cached via `Swatinem/rust-cache`.

Env override for hacking: `OCX_MIRROR_SOURCE=path:<dir> | git+<url>?rev=<SHA> | git+<url>?branch=<name>`. Renderer + bootstrap script honor it.

No submodule. Phase 2 will switch to OCX-self-hosted package.

### D10: Reporting

Per-repo Discord webhook. Silent default. Post only on (a) new green versions or (b) any breakage. No state file, no cross-run dedupe. Aggregator workflow deferred.

Notify message taxonomy:

| Outcome | Post? | Shape |
|---|---|---|
| Nothing new, no failures | silent | — |
| New version(s) all green | post | `📦 <tool>: published <V> (<platforms>)` |
| Partial (some green, some red) | post | `⚠️ <tool> <V> partial — failed: <P> (<reason>). Published: <good-Ps>. Run: <URL>` |
| All-failed | post | `❌ <tool> <V> failed all platforms — see <URL>` |

### D11: Stateless renotify

Drop the state.json idea from earlier draft. `has_new` gate already prevents re-notify spam: if nothing new, smoke does not run, notify does not fire.

### D12: Pipeline status model (per version)

| Status | Meaning | Cascade behavior |
|---|---|---|
| `published` | All declared platforms green + pushed | Full cascade incl. `latest` if newest |
| `partial` | Some green + pushed, some red (not pushed) | Cascade among present platforms; **never `latest`** |
| `failed` | No platforms passed | No manifest pushed at all |
| `skipped-existing` | Already mirrored (set-diff hit) | No action |
| `skipped-executor` | Declared platform has no executor available | Treat as red for that tile |

## Open Architect Calls (Decide With Options + User Feedback)

Each call requires explicit decision in architect output. Present options, recommend one, request user confirmation before locking in.

### A1: `ocx package test` flag reconciliation

`--keep` and `--output` are `conflicts_with` in clap today (per explorer #2). Phase-1 linux flow needs materialize-then-exec-docker.

Options:
- **(a)** Relax `conflicts_with`: allow `--output --keep` together (materialize to dir, don't auto-delete, then exec test command in same invocation)
- **(b)** New flag (e.g. `--materialize-only`) + a separate `ocx package run` for exec
- **(c)** Keep current clap rules; pipeline uses a different command path (does `ocx package install` exist? explorer found only install via symlinks; needs investigation)

Suggested recommendation: (a) — lowest API churn, matches user's pseudo-code shape. Architect should confirm or push back.

### A2: Renderer template strategy

- **(a)** Templates baked into ocx-mirror binary via `include_str!` (cargo-dist style)
- **(b)** Templates loose in `crates/ocx_mirror/templates/` dir, distributed alongside binary
- **(c)** Per-mirror-repo template override capability

Suggested recommendation: (a). (c) is an escape hatch with high complexity cost — defer to phase 2 if real need emerges.

### A3: Renderer drift detection

- **(a)** `ocx-mirror generate ci --check` exits non-zero if generated files differ from what config would produce (run in CI as a step)
- **(b)** Headers in generated files carry hash of config; mismatch checked at workflow runtime
- **(c)** Both

Suggested: (a). Deterministic, cheap, matches user mental model.

### A4: Bumping ocx-mirror SHA across N mirror repos

- **(a)** Renovate-style bot (`ocx-mirror-bump-bot`) opens PR per mirror repo bumping `ocx_mirror.rev`
- **(b)** Reusable workflow in `ocx-sh/.github` consumed by all mirrors — single bump propagates
- **(c)** Hybrid: per-mirror SHA pin (reproducibility) + reusable workflow as emergency floor

Suggested phase 1: (a) — keeps per-mirror reproducibility. (c) for phase 2 once breadth justifies tooling complexity.

### A5: Pipeline state file location (re-raise)

D11 says stateless. But cascade may need a persistent record of "what versions have been published with which platform set" for set-diff against already-mirrored.

Options:
- **(a)** Keep stateless. Use registry's existing tag-list as set-diff source (matches current ocx-mirror behavior per ADR).
- **(b)** Commit `.ocx-mirror/state.json` on default branch for fast reads.
- **(c)** Workflow artifact (90-day retention).

Suggested: (a). Confirms D11. Re-raised because architect may see issues.

### A6: Required-check policy in mirror-repo template

Static matrix → stable check names (`smoke (linux/amd64, ubuntu:24.04)`). Should the template set these as required-checks in branch protection?

- **(a)** Yes — block merge to default branch on red smoke
- **(b)** No — advisory only, maintainer decides per-mirror
- **(c)** Yes for production mirrors, no for experimental

Suggested: (a). Reproducible mirror = the whole point.

### A7: Cascade behavior on partial-version (re-affirm D12)

Architect should explicitly reaffirm or challenge:
- **(a)** Partial cascades among present platforms but never gets `latest` (D12 says this)
- **(b)** Stricter: partial blocks entire version cascade until all green

User leans (a). Confirm.

### A8: `prefix:` field for rosetta-style command wrapping

macOS amd64 needs `arch -x86_64`. Where does this live?
- **(a)** `platforms.<P>.prefix: [...]` in mirror.yml (per-mirror override)
- **(b)** Hardcoded default in renderer's executor table (no mirror config needed)
- **(c)** Default in table, overridable per-mirror

Suggested: (c). Sensible defaults, escape hatch when needed.

### A9: Container shell defaults

- **(a)** Per-container `shell:` field, with sensible defaults (`alpine → sh`, `ubuntu/fedora/debian → bash`, etc.)
- **(b)** Renderer infers from image name pattern (fragile)
- **(c)** Always require explicit `shell:`

Suggested: (a).

### A11: How ocx enters the linux container

Phase 1 linux: ocx runs inside container (D8 corrected). Three install strategies:

- **(a)** Mount host's ocx binary: `-v $(which ocx):/usr/local/bin/ocx:ro`. Fast, offline. Requires musl-static ocx (else breaks across alpine ↔ ubuntu libc).
- **(b)** Curl install inside container: `curl -fsSL ocx.sh/install.sh | sh` as first command. Works any libc. Adds 5–30s per leg. Needs internet from runner.
- **(c)** Pre-baked test images at `ocx-sh/test-runner-<image>:<ocx-ver>`. Fast, no per-run install. Image-maintenance burden.

Phase 2 (qemu + foreign arch) forces (b) or (c) — host's binary won't run under emulated arch.

Suggested phase-1 recommendation: **(a) if ocx is already musl-static, otherwise (b)**. Architect should investigate ocx build profile and decide. (c) is escape hatch for performance-critical mirrors.

Sub-decision: is ocx musl-static today? If not, is making it so blocking on this design or independent? Investigate via grep on `crates/ocx_cli/Cargo.toml` and CI release config.

### A10: GHA service-containers vs manual `docker run`

GHA's `container:` job-level config locks the **entire** job into a container — can't split "ocx on host + test in container" cleanly.

- **(a)** Manual `docker run` in step (current sketch)
- **(b)** Two-job pattern: prepare on host, smoke in container, artifact handoff between
- **(c)** Single container job with ocx installed into the container at start

Suggested: (a). Single-runner-pass keeps things tight.
